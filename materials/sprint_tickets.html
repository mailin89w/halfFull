Track A — Eval & Evidence

Proof that the system works. The most important track for capstone credibility.

T2 Must ship 0.5 day
Run Full Pipeline Eval + Publish Metrics Report
Produce the one artifact reviewers will ask for — concrete evidence the system works, layer by layer. Right now scoring_results.json exists but has never been turned into a clean capstone-grade summary.
▼
Definition of Done
  • python evals/run_eval.py --n 600 --output evals/cohort/ completes cleanly
  • Report includes: ML top-1 / top-3 per condition, Bayesian delta, parse success rate, hallucination rate, over-alert rate on healthy profiles
  • evals/cohort/final_eval_report.md saved — includes a passed/failed DoD table and top-3 failure examples per condition
  • Numbers pass: top-1 ≥ 70%, hallucination < 5%, parse success ≥ 95%, over-alert < 10%
Where to start
evals/run_eval.py is the entry point — flags --layer 1, --n 600, --output are already wired. The metrics_aggregator.py:to_markdown() function already generates the report format. Run it, check the output, fix any import errors, then add the DoD pass/fail table to the top of the generated markdown.
T3 Must ship 0.5 day
Failure Taxonomy: Classify and Document Bad Outputs
A clean accuracy number is less credible than a table of failure types. Knowing your system's failure modes is the difference between a student project and a mature AI system.
▼
Definition of Done
  • 15+ real failure examples collected from the eval run
  • Classified into buckets: overclaiming · wrong_condition_first · sparse_data_overconfident · hallucinated_condition_id · vague_next_steps
  • Saved as evals/cohort/failure_taxonomy.md — each bucket has 2–3 examples and a proposed fix
  • One slide-ready summary line per bucket: "X% of failures were type Y, addressed by Z"
Where to start
After T2 runs, filter scoring_results.json for profiles where top1_correct: false. The scoring_engine.py:score_profile() already captures parse success and hallucination flags — extend it to tag a failure_type field. Sort by failure type and write up examples manually.

Track B — Trust & Safety

Prove the safety layer holds. Write the responsible AI docs almost no student project has.

T4 Must ship 1 day
Safety Regression Suite (Red-Team Test Pack)
The safety layer exists in the architecture — but there's no reusable test pack proving it holds under adversarial inputs. A judge will try to break it during the demo.
▼
Definition of Done
  • evals/safety/red_team_cases.json contains ≥ 20 test cases across 4 categories: diagnostic_language, false_reassurance, emergency_symptoms, sparse_data
  • Script evals/safety/run_safety_eval.py runs all cases through /api/safety-rewrite and checks for guardrail violations
  • Pass/fail table saved as evals/safety/safety_eval_results.md
  • Zero violations in the emergency_symptoms category (hard requirement)
Where to start
Start with frontend/app/api/safety-rewrite/route.ts — the current rules are on lines 8–18 (Groq + Llama-3.1-8b-instant, temp 0.2). Write the 20 test cases as JSON first, then build the runner that calls the endpoint and checks the response for banned phrases like "you have", "you are diagnosed with", "you definitely". The Groq API key is already wired in the frontend env.
T5 Must ship 1 day
Model Card + AI Risk Matrix + "What HalfFull Doesn't Do"
Three responsible AI documents in one day. Each takes ~60 min to write. Together they signal the kind of AI PM thinking that's almost never seen in student capstones.
▼
Definition of Done
  • materials/model_card.md — training data (NHANES 2017–2020), 11 conditions, AUC range, known demographic gaps, intended use, out-of-scope use
  • materials/ai_risk_matrix.md — table: risk type × mitigation in place × gap remaining. Covers: hallucination, bias, overreliance, data sparsity, demographic skew
  • materials/what_halffull_does_not_do.md — plain-language, non-technical: not a diagnosis · not a replacement for a doctor · not validated outside US populations · not for emergencies · all outputs are probabilistic
  • All three linked from README.md
Where to start
models_normalized/*_metadata.json files have all AUC / precision / recall numbers — read these first, copy the range into the model card. evals/cohort/optimization_report.md already documents the known iron_deficiency female bias — quote it directly in the risk matrix. Write the model card first; the other two documents flow naturally from it.

Track C — Demo & Presentation

Guarantee a successful walkthrough. Align the slides with the real system, not the original MVP spec.

T6 Must ship 1 day
Demo Fallback Architecture + Demo Video
If Modal or Railway goes down mid-demo, we need a guaranteed path to a complete walkthrough. The fallback engine already exists in mockResults.ts — it just needs to be surfaced visibly and tested.
▼
Definition of Done
  • Fallback path tested: manually kill HF_ENDPOINT_URL env var and confirm the app completes gracefully with a clearly labelled "offline mode" UI state
  • Offline mode banner visible in the results screen (not a silent fallback)
  • Demo video recorded: 3–5 min screencast of the full user journey — start → assessment → clarify → processing → results → doctor kit PDF
  • Video saved/linked in materials/demo_video_link.md
  • One "known failure modes + fallback" note added to the presentation deck
Where to start
frontend/lib/mockResults.ts is the fallback engine — it already exists. Check frontend/app/api/deep-analyze/route.ts to confirm the error branch calls it and returns a clearly labelled response. Add a yellow "Running in offline mode" banner to results/page.tsx when the response includes source: "fallback". Record the video last, after T2 eval fixes are merged.
T7 Must ship 0.5 day
Prompt Version Changelog + Experimentation Matrix
Prompts are currently hardcoded strings inside API routes with no version history. A prompt changelog is one of the highest AI PM signals and takes half a day to backfill.
▼
Definition of Done
  • Prompts extracted from deep-analyze/route.ts and safety-rewrite/route.ts into named constants in frontend/src/lib/prompts.ts (e.g. MEDGEMMA_SYSTEM_V3, SAFETY_REWRITE_V2)
  • materials/prompt_changelog.md — version table: what changed, why, what eval evidence justified each change
  • materials/prompt_experiments.md — lightweight comparison of 3 variants × dimensions (usefulness, tone safety, parse success, hallucination). Populated with real or manually-scored examples.
Where to start
Run git log -p -- frontend/app/api/deep-analyze/route.ts to reconstruct prompt history from git. Move current prompts to frontend/src/lib/prompts.ts first — then write the changelog backward from the current version. The experimentation matrix doesn't need to be run programmatically; a manually scored table of 3 variants is enough.

Interview Q&A Doc + Architecture-to-Slides Alignment

Track D — High-Signal Product Additions

Features that make the architecture diagram come alive in the UI. Pick 2–3.

T9 High signal 1 day
Visible Reasoning Trace on Results Screen
Turn the architecture diagram into a live UI feature. Show users (and reviewers) exactly how the system went from answers to conclusions — step by step. Practices multi-step CoT/ReAct display, currently absent from the frontend.
▼
Definition of Done
  • Results screen has a collapsible "How we reached this" panel per condition card
  • Panel shows 4 steps with real numbers: ML score → Bayesian update → cluster fingerprint → MedGemma synthesis
  • Example: "ML: Anemia scored 0.71 (threshold 0.45) · Bayesian: fatigue + cold intolerance → 0.83 · Cluster: 68% of neighbours flagged low ferritin · Synthesis: MedGemma"
  • Frontend-only change — no new API endpoints needed
Where to start
Check what /api/bayesian-update and /api/deep-analyze already return — the Bayesian trace is in the response. Add a <ReasoningTrace /> component to frontend/src/components/results/DiagnosisCard.tsx that renders the trace collapsibly. The data is already flowing — this is purely a display feature.
T10 High signal 1 day
Confidence Orchestration Layer
Replace raw probability numbers with a user-facing confidence signal that combines all evidence sources. Closes the "confidence-weighted urgency" backlog item with a concrete, tested implementation.
▼
Definition of Done
  • Each condition card shows one of three tiers: Low confidence · Moderate confidence · High confidence with a one-line explanation
  • Score is computed from: ML posterior + Bayesian update magnitude + cluster support % + data completeness (did user submit labs?)
  • Formula documented in materials/confidence_score_design.md
  • No new backend needed — computed in frontend/src/lib/confidence.ts from existing response fields
Where to start
frontend/src/lib/types.ts defines the result schema — read it first to see what fields are available. Create frontend/src/lib/confidence.ts with a computeConfidence(result, completeness) function. Use it in DiagnosisCard.tsx. For completeness, count non-null answers from useAssessment.ts state and pass it down.
T11 High signal 0.5 day
Citation Tags + Provenance Panel
Close the gap between "we claim evidence-backed reasoning" and "we can show where each claim came from." Even without full RAG, showing the pipeline inputs is a strong trust signal.
▼
Definition of Done
  • Each insight in the results screen is tagged with its source: [ML model] · [Bayesian update] · [Cluster] · [MedGemma]
  • A collapsible debug/demo panel shows the MedGemma input context — what signals were passed in — making the pipeline visible without requiring full RAG
  • If RAG is implemented (T13): panel shows top-2 retrieved chunks with source title
Where to start
frontend/app/api/deep-analyze/route.ts — log the full prompt context sent to MedGemma and return it in the response as a context field. Add a <ProvenancePanel /> component to the results screen, collapsed by default. This surfaces what the system already knows — no new logic required.
T12 High signal 1 day
"Not Enough Information" States + NHANES Percentile Display
Two UX features that show AI product maturity and leverage the depth of the NHANES dataset. Designing graceful uncertainty is an advanced AI UX skill almost no student project demonstrates.
▼
Definition of Done
  • If user completes < 60% of questions: results screen shows a calibrated uncertainty banner — "Based on what you've shared, we can flag early signals but can't rank conditions reliably without lab values"
  • For at least 3 key lab values (ferritin, TSH, hemoglobin): results show a percentile label — "Your ferritin is in the 12th percentile for adults aged 30–50 in NHANES"
  • Percentile lookup is a static JSON file derived from data/processed/nhanes_merged_adults_final.csv

Percentile file: run a one-time Python script on nhanes_merged_adults_final.csv to compute decile tables per lab × age_group × sex, save to frontend/src/data/nhanes_percentiles.json. Add a <PercentileTag /> component alongside each lab value in the results.

Track E — AI Skills Depth

The two most portfolio-valuable additions. Close the RAG gap and run a real experiment.

T13 High signal 1.5 days
Minimal RAG on PubMed Abstracts
The original plan and skill map both committed to ChromaDB RAG on PubMed abstracts. The architecture diagram shows a "RAG on knowledge corpus" box. Right now we use MedGemma directly with no vector store. This ticket closes that gap or gives us a clear documented reason why we chose not to.
▼
Definition of Done
  • 10–15 PubMed abstracts per condition (top 11 conditions) embedded and stored in ChromaDB or Pinecone free tier
  • On each /api/deep-analyze call, top-2 relevant chunks retrieved and injected into MedGemma context
  • Each retrieved chunk's title visible in the T11 provenance panel
  • rag/ directory in repo: ingest.py, retrieve.py, data/abstracts/*.txt
Where to start
frontend/src/lib/medgemma.ts is where MedGemma is called — add a retrieveContext(conditions) call before the prompt is assembled. Use OpenAI text-embedding-3-small (smallest, cheapest) or chromadb's default embeddings. PubMed abstracts: search PubMed for the 11 condition names, export 10–15 abstracts each as .txt files. Total ingest: ~110 documents, well within any free tier.
T14 High signal 1 day
A/B Prompt Experiment on Synthetic Cohort
We're possibly the only team in the cohort that can actually run a real LLM A/B test at scale — because we built the eval harness. Using it for a prompt experiment is a unique, demonstrable AI PM skill.
▼
Definition of Done
  • Variant A: current MedGemma zero-shot prompt
  • Variant B: few-shot prompt with 3 curated clinical examples added to the system prompt
  • Both run on the same 50-profile subset (10 conditions × 5 profiles each)
  • Comparison table: parse success · hallucination rate · output specificity (manually scored 1–3) · tone safety (pass/fail via safety-rewrite)
  • Saved as evals/ab_test/results.md with a clear conclusion: which variant to ship and why
Where to start
Depends on T7 (prompts extracted to lib/prompts.ts). Copy evals/run_eval.py, add a --prompt-variant A|B flag that swaps the system prompt. Run on 50 profiles, not 600 — keep it tractable. The few-shot examples should be 3 real condition summaries you've already seen the system produce well.

Track F — Stretch

Only if Tracks A–C are fully done. Each adds a distinct interview story.

T15 Stretch 1 day
Anomaly Detection via KNN Cluster Layer
Flag labs that are statistically unusual within a person's cluster neighbourhood. Practices anomaly detection — not in the original skill map — with a concrete, impressive demo moment.
▼
Definition of Done
  • For each flagged condition, results show: "Your CRP is elevated relative to 87% of your nearest neighbours in NHANES"
  • Uses existing scripts/cluster_knn_inference.py (K=50, 15% consensus threshold already implemented)
  • Results surfaced in the "Extended lab flags" section of the results screen
  • Output added as an additional field in the /score API response
Where to start
scripts/cluster_knn_inference.py already computes neighbourhood lab abnormality scores. data/processed/cluster/ has the cluster assignments and UMAP artifacts. Wire the inference script output into api/server.py as an additional field in the /score response. Then display it in the results frontend.
T16 Stretch 0.5 day
Docker + GitHub Actions CI
Show shipping discipline. The Docker gap is explicitly called out in the skill map. A Dockerfile + passing CI check in the repo header is a low-effort, high-visibility signal for any technical reviewer.
▼
Definition of Done
  • Dockerfile in /api/ wraps FastAPI + model files, runs uvicorn api.server:app
  • docker build and docker run tested locally
  • .github/workflows/ci.yml runs pytest api/ on every push to main
  • README.md updated with docker run instructions and a passing CI badge
Where to start
api/requirements.txt and the Procfile define the runtime — use python:3.11-slim as the base image. Models are large (~100MB each) — use .dockerignore to exclude data/, notebooks/, and evals/cohort/. The GitHub Action is 10 lines: checkout, setup-python, install requirements, run pytest.
T17 Stretch 1 day
Longitudinal Tracking UI (Returning User)
Reframe the product from "one-time quiz" to "health monitor over time." A strong product story that requires no backend — just localStorage — and opens a genuinely different product narrative for the presentation.
▼
Definition of Done
  • If localStorage contains a previous result, a "Compare to last time" banner appears on the results screen
  • Energy score delta shown: "↑ 12 points since Feb 2026"
  • Condition status change shown per condition: "Anemia risk: was High → now Moderate"
  • A simple /history page lists past assessments by date with a mini score chart
Where to start
frontend/hooks/useAssessment.ts manages localStorage state. Extend it to save a timestamped result snapshot on completion (key: halffull_history, value: array of {date, scores, conditions}). Add a <HistoryBanner /> component to results/page.tsx that renders the delta if a previous result exists. No backend required.
